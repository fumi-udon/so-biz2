# Söya Order System — 技術契約 V4（Phase 4 POS 前提）

*目的: 実装・マイグレーション・Echo・Livewire 間で **同一スペル（命名）** を厳格に保ち、「後で改名する」という技術的負債の発生を禁止する。あわせて §5（Revision Contract）・§6（UI 状態機械）・§7（ゼロロスト伝達）で、**サーバー権威・楽観的ロック・スタッフ UI の安全・KDS への到達** をチーム全体の契約として固定する。Phase 5 以降は §9（インフラ SLO / Outbox）・§10（KDS Pull / Keyset）・§11（チケット単位 `line_revision`）を **KDS・運用の法** として追加する。*

## 5. 楽観的ロックと条件付き更新（Revision Contract）

*原則: **サーバー権威**。クライアントの画面は常に「キャッシュ」であり、危険な状態遷移（会計・確定など）は **DB 上の現在値とリクエストに添付したリビジョンが一致する場合のみ** 許可する。*

### 5.1 バージョン指標（チーム内で 1 系統に統一）

| リソース | 推奨列名（論理） | 型（契約） | 意味 |
|----------|------------------|------------|------|
| `table_sessions` | `session_revision` | 非負整数（単調増加） | 当該セッションに関する **ホール側ドロワーに影響する変更** のたびにインクリメントする。新規注文行の付与、会計、セッション状態遷移、スタッフ操作による注文状態変更など、**契約で列挙したイベント**ごとに更新する。 |
| `orders`（`PosOrder`） | `order_revision` | 非負整数（単調増加） | **注文ヘッダ単位**の変更（行追加・状態遷移・金額再計算後の確定など）のたびにインクリメントする。 |

*代替案: 初期段階で物理列が無い場合でも、**最終的には上記の整数リビジョンへ収束させる**。`*_revision` が無い状態で `updated_at` のみを比較するのは **同一秒内の複数更新で衝突しうる** ため、本番運用では非推奨とする。*

### 5.2 必須付与ルール（API）

| 操作カテゴリ | リクエストに含めるリビジョン | 備考 |
|--------------|------------------------------|------|
| セッション単位（例: **Cloture**、セッション全体に効く操作） | `session_revision`（必須） | 対象 `table_session_id` の現在値と一致すること。 |
| 注文単位（例: **Confirm / Recu 相当の単発確定**） | `order_revision`（必須）に加え、文脈によって `session_revision`（推奨） | 二重の門番により、**注文だけ古い／セッションだけ新しい** といった不整合を検出しやすくする。 |

*HTTP 表現: JSON ボディに `session_revision` / `order_revision` を載せることを標準とする。`If-Match` を併用する場合は **同一セマンティクス** で解釈し、二重定義を禁止する。*

### 5.3 競合時の応答（HTTP `409 Conflict`）

- **ステータスコード:** `409 Conflict` 固定。
- **Content-Type:** `application/json`。
- **標準ボディ（必須キー）:**

| キー | 型 | 意味 |
|------|-----|------|
| `message` | string | 人間可読（スタッフ向け短文案内可）。 |
| `code` | string | 機械可読。固定: `REVISION_CONFLICT`。 |
| `context` | object | 下表の **Context オブジェクト**。 |

**`context` オブジェクト（必須キー）**

| キー | 型 | 意味 |
|------|-----|------|
| `resource` | string | `table_session` \| `pos_order` など、競合したリソース種別。 |
| `id` | int | 対象レコードの主キー。 |
| `current_revision` | int | サーバー上の **真の** 現在リビジョン。 |
| `client_sent_revision` | int | クライアントが送ったリビジョン。 |

*クライアントは `409` 受信後、**必ず最新状態を再取得（Pull）** してから操作を再開する。自動リトライで同一リクエストを繰り返すことは禁止（意図しない二重会計の温床になるため）。*

### 5.4 関連ドキュメント

- 業務上の「Human filter」と KDS への送付タイミングは [order-system-ver1.md](./order-system-ver1.md) に従う。本 Revision Contract は **その境界を技術的に強制するための装置** と位置づける。

## 6. UI 状態機械（UI State Machine）

*対象: ホール POS の **TableActionHost 等、スタッフが単一の危険ボタンを操作する UI**（Livewire / Alpine.js）。*

### 6.1 アクションの 5 状態

| 状態 | 定義 | 許可される UI 操作 |
|------|------|---------------------|
| `idle` | サーバーへの当該ミューテーションは **未送信**。 | ボタン有効。 |
| `in_flight` | 当該アクションの **1 リクエストが進行中**（応答待ち）。 | **同一アクションの再送信は禁止**（ボタン disabled + 視覚的 loading）。別アクションを許すかは画面ポリシーで定めるが、**Cloture と Confirm を同時 in_flight にしない**ことを推奨。 |
| `success` | サーバーが **成功**を明示的に返し、クライアントが確証を得た。 | 通知表示。必要に応じタイル再取得・`loadSessionData` 等。 |
| `failed` | サーバーが **失敗**を明示的に返した（4xx/5xx でボディに理由がある想定）。 | エラー表示。**同一冪等キーで再試行可能**（バックエンド契約に従う）。 |
| `unknown` | **タイムアウト・接続断・応答なし** など、成功・失敗が判定できない。 | **長時間スピナーのみ**は禁止。必ず「結果不明」表示と **状態照会（Pull）** または **明示的な再試行** の導線を出す。 |

### 6.2 `in_flight` 中の二重送信防止

- **同一アクション・同一意図** に対しては **単一フライト**（同時に 1 リクエストのみ）。
- バックエンドが冪等キーを要求する場合、クライアントは **1 操作あたり 1 冪等キー** を生成し、`unknown` に落ちるまで **同一キーでの再送を自動では行わない**（再試行はユーザーの明示操作または照会後の `idle` 復帰後）。

### 6.3 `unknown`（タイムアウト）時の UX ルール

1. **タイムアウト ≠ 失敗**。画面上は `unknown` と表示し、スタッフが「通ったか分からない」パニックにならない文言にする。
2. **第一の回復:** **状態照会** — セッション／注文の **現在リビジョンと一覧** を GET（または Livewire の read-only 再取得）し、操作が反映済みか判定する。
3. **第二の回復:** 照会の結果、未適用なら **idle** に戻し、**同一冪等キーで再試行** を許可（バックエンド契約と整合）。
4. **Echo / プッシュ** による `loadSessionData` と競合する場合は、**別紙の調停**: `in_flight` 中はプッシュ起因の全置換リロードを **キューに積み、同一バーストは最後の 1 回にまとめる（coalesce）**。**アクション完了後に 1 回** まとめて反映する（詳細は実装タスクで本契約に準拠）。

## 7. KDS への確実なイベント伝達（Zero-Lost Event Delivery）

### 7.1 プッシュの位置づけ

- **Echo / Pusher による配信は「通知」であり、到達保証をしない。** 遅延・欠落・重複は **常に前提とする**。
- **真実の源泉は常に DB**（およびそのトランザクション境界）。KDS 画面の最終的な正しさは **Pull による追いつき** で担保する。

### 7.2 Transactional Outbox（必須パターン）

| 項目 | 契約 |
|------|------|
| 目的 | **「DB に書いた事実」と「外部へ飛ばすべき通知」を同一コミットで整合**させる。 |
| 実装方針 | アプリの同一トランザクション内で、(1) 業務テーブル更新、(2) **`outbox` 相当テーブルへの行挿入** をコミット。別プロセス（キューワーカー）が outbox を読み、Pusher 送信成功後に **処理済み** とマーク。 |
| 失敗時 | 送信失敗は **リトライ**。ワーカー停止時は **outbox 滞留メトリクス** で検知する。 |
| 禁止 | **「先に Pusher 成功を待ってから DB コミット」** あるいは **DB なしのプッシュのみ** を本番の唯一の経路とすること。 |

*物理テーブル名・列名は実装時に定義するが、**冪等キー**（下記）を保持できること。*

### 7.3 冪等な消費（Idempotent Consumer）

- KDS 側（または中継サービス）は、各イベントに付与される **`dedupe_key`**（例: `shop_id` + `pos_order_id` + 遷移名 + `order_revision` など、**重複してはならない単位で一意**）を記録し、**既に処理済みなら再送を無視**する。
- **少なくとも一度（at-least-once）** 配送を許容し、**処理側が冪等**であることが前提。

### 7.4 Pull 型バックストップ（必須）

| 項目 | 契約 |
|------|------|
| 目的 | プッシュ未達時でも **DB と KDS の表示が最終的に収束**する。 |
| トリガー | **定期ポーリング** および **画面フォーカス／復帰時** の少なくとも二経路を持つ。 |
| API 要件 | **店舗スコープ**（`shop_id`）および **カーソル**（例: `(updated_at, id)` の辞書式順序）による **差分取得**。同一秒内の複数行は **`id` でタイブレーク**することを明記する。 |
| 応答 | キッチンが必要とする **確定済み注文／行** の一覧または差分。プッシュのペイロードと **同一の論理モデル**（状態・ID・リビジョン）で解釈可能であること。 |
| 禁止 | Pull と Push で **別ルールのフィルタ** を適用し、表示が分岐し続けること。 |

### 7.5 `broadcastWith` との拡張（後方互換）

- §3.2 の ID ベースペイロードに、実装フェーズで **`session_revision` / `order_revision` / `dedupe_key` / `event_id`** を追加する場合、**既存キーを壊さない** 前提で **追加のみ** とする。ブロードキャスト名（`broadcastAs`）の変更は **V5.0 改定**（§12）に準ずる。

## 9. インフラストラクチャと Outbox の確約（Infrastructure SLO）

*目的: §7.2 の **Transactional Outbox** を **ホールの HTTP レイテンシから完全に切り離す**。同期キュー・同期外部 API 待ちによる現場停止を契約上禁止する。*

### 9.1 ホール HTTP と外部ブロードキャストの分離（禁止事項）

| 原則 | 定義 |
|------|------|
| **同期 Pusher 禁止** | ホールのミューテーションを処理する **HTTP リクエストのクリティカルパス内** で、Pusher（および同等の外部ブロードキャスト HTTP API）を **完了待ち（await）** してはならない。 |
| **レスポンス優先** | スタッフ向けレスポンス（JSON / Livewire 応答）の生成に、**外部 SaaS の往復時間を含めない**。 |
| **真実の源泉** | 業務状態の確定は **DB コミットまで**。通知は **ベストエフォート**（§7.1）。 |

*解釈: アプリコードが「コミット後に即 `broadcast()` を同一リクエスト内で同期的に呼び、その完了を待つ」設計は、インフラ次第で **Cloture 等が外部遅延でフリーズ**するため **本契約に違反**する。*

### 9.2 本番キューとワーカー（必須要件）

| 項目 | 契約 |
|------|------|
| **`QUEUE_CONNECTION`** | 本番環境では **`database` または `redis`** とする。`**sync` を本番の最終形としない。** |
| **非同期ワーカー** | **`php artisan queue:work`（または同等）の常駐プロセス**を **インフラ必須要件** とする。デプロイ手順・Runbook に明記する。 |
| **Outbox の帰属** | §7.2 における **outbox 行のドレイン（Pusher 送信・処理済み化・失敗リトライ）** は **ワーカー専用** とし、**Web リクエストプロセスが担当しない**。 |

*補足: 開発・CI で一時的に `sync` を用いることは許容するが、**本番 SLO・負荷試験・障害想定は必ず §9.2 準拠の構成で検証する**。*

### 9.3 §7.2 との関係

- Outbox パターンの **「コミット」と「送信」の責務分離** は、**本 §9 によりインフラ面で保証**する。ワーカー未稼働は **Outbox 滞留**として監視し、運用事故として扱う。

## 10. KDS Pull API のスケーラビリティ契約（Keyset Pagination）

*目的: KDS 端末からの差分 Pull が **インデックス効率的**であり、**OFFSET による線形悪化**を起こさない。*

### 10.1 カーソル（必須）

- クライアントが送るカーソルは **必ず `(updated_at, id)` の辞書順** で比較する（§7.4 と同一論理）。
- 同一 `updated_at` 秒内に複数行が存在しうるため、**第二キー `id` によるタイブレークは必須**（取りこぼし禁止）。

*典型的な「次ページ」条件（意味の例）:*  
`(updated_at > :t) OR (updated_at = :t AND id > :id)`  
※ 実装ではマイクロ秒精度・タイムゾーン・サーバー時刻の一貫性を別紙で固定する。

### 10.2 禁止: `OFFSET` ベースのページネーション

| 禁止 | 理由 |
|------|------|
| **`OFFSET` による Pull** | 深いページで **フルスキャンに近いコスト**が発生し、ピーク時に 5 台以上の iPad が同時再接続した際に DB を破壊する。 |

### 10.3 許可: Keyset（シーク）のみ

- Pull API は **Keyset Pagination のみ** を許可する。
- WHERE 句は **範囲走査に使う複合インデックスの先頭列と整合**させること。

### 10.4 複合インデックス（指針例）

*実際の WHERE / JOIN に合わせて最終確定する。以下は **例** であり、**スキーマ確定後にマイグレーションで固定**する。*

| 指針 | 内容 |
|------|------|
| 左端 | **`shop_id`** 等、**必ず絞る**スコープ列 |
| 状態 | KDS が列挙する **`status`（複数可）** がクエリに含まれるなら、インデックスに含める |
| カーソル | **`updated_at`, `id`** を **インデックス末尾**に含め、ソート順と一致させる |

*例（論理名）:* `(shop_id, status, updated_at, id)` — **Explain とピーク QPS で検証**し、不足なら改定（§12）。*

### 10.5 補足: 負荷平準化

- クライアントは **ジッター付き再試行**・**最大件数上限** を実装し、再接続直後の **サンダリングヒード**を避ける（別紙 UX 契約）。

## 11. オーダーラインの楽観的ロック（Ticket-level Revision）

*目的: 複数シェフが **同一オーダーライン（チケット）** に対して同時操作したとき、**二重遷移を論理的に不可能**にする。ホールの `session_revision`（§5）と対になる **行単位の権威** を定義する。*

### 11.1 スキーマ（必須）

| テーブル | 列名（論理） | 型（契約） | 意味 |
|----------|--------------|------------|------|
| `order_lines` | `line_revision` | 非負整数（単調増加） | 当該行の **KDS 向け状態・表示に影響する更新** のたびにインクリメントする。実装では `line_revision` と同名である必要はないが、**チーム内で 1 指標に統一**する。 |

*代替列名を用いる場合も、**本契約の意味（チケット単位楽観ロック）** を満たすこと。*

### 11.2 リクエスト必須フィールド

| 操作 | 必須 |
|------|------|
| 「調理開始」「提供済」等、**行の業務状態を変える** KDS ミューテーション | `order_line_id` に加え **`expected_line_revision`**（クライアントが表示時点で保持した値） |

### 11.3 サーバー側の条件付き更新（必須）

- 状態遷移は **`UPDATE order_lines SET ..., line_revision = line_revision + 1 WHERE id = ? AND line_revision = ?`**（または同等の **1 行・原子的** 条件付き更新）で行う。
- **一致しなければ更新行数 0** → **競合**として扱う。

### 11.4 競合時の応答（§5 と同型）

- **HTTP ステータス:** `409 Conflict`（API の場合）。Livewire 等の場合も **同一の意味**で `REVISION_CONFLICT` を返す。
- **ボディ:** §5.3 の **`message` / `code` / `context`** に準ずる。`context.resource` は **`order_line`**、`context.id` は **オーダーライン ID**、`current_revision` / `client_sent_revision` は **`line_revision`** の値とする。
- **クライアント動作:** **再 Pull（差分取得）** を促し、**自動の無制限リトライは禁止**（§5 と同様）。

### 11.5 UX（指針）

- KDS フロントは §6 の **UI 状態機械**に準拠し、`409` 受信後は **トースト + 一覧再同期（Pull）** とする。
