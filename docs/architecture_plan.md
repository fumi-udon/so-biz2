# 🏛️ Söya Order System: Master Architecture & Execution Plan (v2)

## 1. プロジェクトの基本理念 (The "Stone Bridge" Philosophy)
本プロジェクトは、ゲスト（顧客）、ホール（POS）、キッチン（KDS）をつなぐミッションクリティカルなオーダーシステムである。「石橋を叩いて渡る」を絶対の原則とし、以下のアーキテクチャ方針を貫徹する。

* **Single Source of Truth (単一の真実):** システムの全状態（テーブル状況、注文内容、金額）の決定権はバックエンド（DB）およびPOSが持つ。
* **Immutability (不変性):** 確定（Confirmed）したオーダーの金額・内容は絶対に変更不可とする。修正は「Void（取消）」と「新規追加」のトランザクションで表現する。
* **Idempotency (冪等性):** 通信エラー、リロード、ダブルタップなど、あらゆるクライアントの異常挙動に対して二重決済や二重オーダーを100%防ぐ。
* **Test-Driven Foundation (テスト駆動の土台):** 新機能実装前に、必ずエッジケース（競合・瞬断）を含めた自動テストを完遂させる。

## 3. 次期要件定義: POS / テーブルシステム (Phase 4)
ホールスタッフが直感的に操作できる「16卓固定配置（将来拡張可）のタイル型ダッシュボード」。POS画面は**「Filamentのカスタムページ」としてフレーム内に内包**し、二重管理を避ける。

### 3.1. 状態管理とViewModel（チラつき・乖離の防止）
テーブルは複数の注文（PlacedとConfirmedの混在など）を持つため、「卓の色」を決定するロジックは単一の `TableDashboardViewModel` 等に集約し、以下の優先順位で導出する。

* **[優先度 1 / アクション必須] 薄い赤 (Placed):** 卓内に1つでも「未確定（店員の確認待ち）」の新規注文が存在する。
* **[優先度 2 / アクション準備] イエロー (Addition Printed):** 中間伝票が印刷済、または会計依頼が出ている。
* **[優先度 3 / アクション不要] 白 (Cooking/Confirmed):** すべての注文がキッチンへ送信済で、調理・提供待ち。
* **[優先度 4 / 待機] ブルー (Seated):** アクティブなセッションがあるが、まだ注文がない。
* **[優先度 5 / デフォルト] グレー (Empty):** クローズ済み、QRスキャン待ち。

## 4. 未来の要件定義: KDS / キッチンアプリ (Phase 5)

### 4.1. データ正規化の事前定義（技術的負債の回避）
KDSでのポジション分離（Kitchen/Counter）を見据え、Phase 4に入る前に `menu_items` テーブルに `role_category`（enum等）を明確に定義し、JSONフラグによる場当たり的な実装を回避する。

### 4.2. ハイブリッド通信設計と雷群（Thundering Herd）対策
WebSocket（Reverb）は「更新通知」のみに使用し、データはLivewireでフェッチする。本番負荷と信頼性を担保するため、以下のセーフティネットを必須とする。

* **ジッター（Jitter）とDebounce:** `OrderPlaced` イベントを受信した際、全端末が同時にDBへフェッチに行く雷群現象を防ぐため、Livewireのフェッチ処理にランダムな遅延（ジッター）またはDebounceを挟む。
* **Degraded Mode（劣化時挙動）:** WebSocketが切断された場合（あるいはタブがスリープから復帰した場合）の「静かに壊れた画面」を防ぐため、バックアップとして `wire:poll.30s`（30秒に1回の静かな定期フェッチ）を併用する。
* **イベント契約（Contract）:** Broadcastされるイベントは軽量化し、ペイロードは `['table_session_id' => 123]` などのIDのみに留め、認可漏れによる情報漏洩を防ぐ。

## 5. 直近の実行タスク (Execution Step: Phase 2.8)
本フェーズでは、システム境界で発生する「飲食店特有の最悪なエッジケース」をFeatureテストで完全に網羅する。

**実装すべき貫通複合テスト（GuestOrderLifecycleTest.php）:**
1. **セッション・ライフサイクルの基本貫通:** QRスキャン → セッション生成 → カート送信 → テーブル紐付け。
2. **トークン・ミスマッチ防御:** URLの `tenant/table` とペイロード `context` が不一致の場合の拒否（別卓QRの誤スキャン対策）。
3. **トランザクション境界の競合（最重要）:**
   * **「会計（Close）」と「ゲスト追加注文（Submit）」の完全同時発生:** どちらが勝つべきかの仕様を定義し、ロックタイムアウトや二重決済を防ぐこと。
   * **同一セッションでの連続2回送信:** 同一 `idempotency` キーによる二重投稿の完全ブロック。

これらのテストをパスするまで、FilamentのUI作成（Phase 4）やWebSocket（Phase 3）には一切進まない。
